Question 6

It is very difficult to accurately calculate the stack size because:

  • As a programmer, it is never certain how many function calls may be produced (especially recursive functions)
  • **Anything that depends on external input/user input cannot be controlled ** —> the amount of ISRs called can affect the accuracy of calculating stack size
  • Can be unsure of how things are scheduled which makes it uncertain how many tasks may need access to stack at the same time, meaning if the value is not bounded stack size can vary quite a lot
  • Library calls may also cause you not to know what it will require in regards to stack size (*printh library call for example)

Two techniques which can help continuously to keep track of the maximum stack requirement:

  • Can write bit patterns into the space in which the stack can fill up and then “go in” and see how long the pattern is, i.e., where the bit pattern has been overrided
    • This can then show you where the stack “is”, by looking at how far the bit patterns have been “erased”
  • Can “program” a function into your code which can contain just a local variable which is just a pointer and call this function.
    • This gives you the top of the stack as it points to the “last accessed memory”

Question 8

  • SPI: Serial Peripheral Interface
  • UART: Universal Asynchronous Receiver-Transmitter
  • I2C: Inter-Integrated Circuit

UART is much preferred over everything here because it is asynchronous, meaning transmission and receiving can happen in both directions at the same time. Also easy to implement and easy to debug. Downside is only 2 units are able to use at once and it’s slower because you typically use UART over longer distances (wires) than the other two. More reliable

SPI/I2C are master-slave type transmission, where master just goes around and sends messages to the slaves telling them to send ‘me’ the “temperature” (for e.g.,). SPI is faster than the other two so preferable when speed is needed.

I2C is preferable when lots of devices need connecting together. Downside is device needs to be able to send an address (e.g., if you have 100 different temperature sensors, they each need to have the ‘intelligence’ to send a unique address, 100 unique addresses. That’s a lot!)

Question 10

Interrupt and dispatch latency times play a critical role in meeting deadlines in a hard real time system

Interrupt latency is the time it takes for the system to recognise and respond to an interrupt request from a device or process. Dispatch latency is the time it takes for the system to switch from executing one task to another.

If these latencies are not bounded, it means that there is no guarantee that the system will respond to an interrupt or switch to another task within a specified time frame. This can result in missed deadlines, system failure, or even safety hazards in critical systems like medical devices, aerospace control systems, or nuclear power plants.

For example, consider a hard real-time system that controls the speed of a train. If the system does not respond to an interrupt from the track signalling system within a specific time frame, it may not be able to apply the brakes in time to prevent a collision. Similarly, if the dispatch latency is too long, the system may not be able to switch to an emergency task, like applying the brakes, quickly enough to prevent an accident.

In summary, bounded interrupt and dispatch latency times are essential in hard real-time systems to ensure that tasks are completed within their specified deadlines, and to prevent system failures or safety hazards.

Disabling interrupts means the system clock is not updated because the clock in most computer systems relies on interrupts to function accurately. When interrupts are disabled, the clock's accuracy may be affected because the clock depends on regular signals from the hardware timer, which is itself driven by interrupts

When interrupts are disabled, the hardware timer stops generating interrupts, and the system clock can fall out of sync with the actual time. This can lead to time drift, where the system clock runs faster or slower than real-time

Strategies to use to counteract this:

  • Minimise the time interrupts are disabled: If interrupts must be disabled, it is essential to keep the duration of the interruption as short as possible

Ultimately, we want to disable them as briefly as possible, ISRs need to be simple and fast.
