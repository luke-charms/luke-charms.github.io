Watchdog Timers

Autonomous Fault Recovery

Typical embedded systems need operate continuously, but cannot rely on human supervision:

  • system not easily accessible
    • implanted device
    • spaceprobe
  • human supervision is too costly
  • human supervision is too slow

⇒ use a watchdog to supervise the system

—> Endless loops —> Interrupt flood —> Deadlocks from multitasking

Protection

It is important that a run-away program will not accidentally disable the watchdog timer before it times out!

—> There are typically processor specific protection methods - always document the build process!

—> If the watchdog is not always used clearly, its enabling bits at the start of program initialization

—> A good strategy is to count the number of consecutive watchdog resets and reconfigure or disable the watchdog, if the count appears excessive.

Watchdog Timer Selection

When selecting a watchdog timer for your project, consider the required timeout duration and the oscillator accuracy. If a precise timeout is important, a crystal oscillator may be required. Additionally, it is important to choose a timer with a configurable timeout period to match the needs of your system.

Software Integration

To ensure proper integration of the watchdog timer into your system, it is important to thoroughly test its functionality and activation prior to production. Additionally, it is recommended to have a plan in place for handling unexpected reboots caused by the watchdog timer, such as logging and reporting the event.

Watchdogs differ from device to device

  • Some watchdogs can only be enabled, but not disabled
  • Some watchdogs allow an extra delay during the first time out period
    • Account for initialization period
  • Some watchdogs are disabled after a time-out
    • need to be enabled during initialization

Watchdogs have different modes - important to know which mode is active in your system ! (don’t make assumptions when switching between microcontrollers)

Multi-tasking: Monitor Task

  • Monitor task can watch over other tasks
    • This is the only task allowed to restart the watchdog
  • All tasks report their operation by acting on a global set of flags
    • E.g., clearing the flag when task completes
  • Monitor task only restart the watchdog if all tasks have reported success
  • Checks whether all other tasks are running, and then if not, restart the watchdog. Unlike the Toyota example, where the watchdog was reset every time

: Time Scales

Often Tasks run at very different time scales.

- A frequent high priority task may be critical

- A less frequent task will report at a much lower frequency
