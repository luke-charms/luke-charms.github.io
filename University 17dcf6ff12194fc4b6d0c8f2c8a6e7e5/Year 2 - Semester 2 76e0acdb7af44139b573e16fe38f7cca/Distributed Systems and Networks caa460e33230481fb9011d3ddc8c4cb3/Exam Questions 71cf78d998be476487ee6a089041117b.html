Here's how the process unfolds:

  1. PC1 consults its routing table to determine the next hop for delivering packets to the destination IP address and forwards the packet to the default gateway
  1. The default gateway receives the packet from PC1 and examines the destination IP address (192.168.1.4). It compares the destination IP address with its own routing table entries to determine the next hop for the packet.
  1. The default gateway realises that PC2 (192.168.1.4) is on the same local network and directly connected. It performs an Address Resolution Protocol (ARP) request to obtain the MAC address of PC2.

  1. The default gateway sends the Ethernet frame containing the encapsulated IP packet to PC2 on the LAN.
  1. PC2, being on the same local network, receives the Ethernet frame and extracts the encapsulated IP packet.
  1. PC2 checks the destination IP address within the IP packet (192.168.1.4) and matches it with its own IP address. As they match, PC2 determines that the packet is meant for itself.
  1. Finally, PC2 processes the ICMP ping request packet and generates an appropriate response.

Question 3

  1. Window Size: TCP uses a sliding window mechanism to control the flow of data. The window size indicates the number of bytes a sender can transmit before requiring an acknowledgment from the receiver. The receiver advertises its window size, indicating how much data it can accept. The sender adjusts its transmission rate based on the receiver's window size, ensuring that it doesn't overwhelm the receiver with more data than it can handle.
  1. Congestion Control: TCP utilises congestion control algorithms to prevent network congestion. When congestion is detected, typically through packet loss or increased network delays, TCP reduces its sending rate to alleviate the congestion. One widely used congestion control algorithm is TCP Reno, which employs mechanisms like Slow Start, Congestion Avoidance, and Fast Retransmit/Fast Recovery to regulate the flow of traffic and maintain network stability.
  1. Explicit Congestion Notification (ECN): ECN is an extension to TCP that allows network devices to notify endpoints of impending congestion before packet loss occurs. When a congested network device encounters congestion, it marks packets with an ECN flag instead of dropping them. The receiving TCP endpoint interprets the ECN flag and adjusts its transmission rate accordingly, helping to prevent congestion before it escalates.
  1. Flow Control: TCP implements flow control to ensure that a fast sender doesn't overwhelm a slow receiver. The receiver specifies its receive window, indicating the amount of data it can buffer. The sender regulates its transmission rate based on the receiver's window, preventing data overflow at the receiver's end.
  1. Retransmission and Timeout: TCP uses retransmission and timeout mechanisms to handle packet loss. If a sender doesn't receive an acknowledgment for a transmitted segment within a certain time (timeout period), it assumes that the packet was lost and retransmits it. By retransmitting lost packets, TCP ensures reliable delivery of data.

In contrast, IPv6 addresses are 128-bit, which allows for a practically unlimited number of unique addresses—approximately 340 undecillion (3.4 x 10^38). This expanded address space is essential to accommodate the growing number of devices and the increasing demand for IP addresses in the era of the Internet of Things (IoT).
  1. Improved Addressing and Auto-configuration: IPv6 incorporates features that simplify address management and facilitate automatic address configuration. One such feature is Stateless Address Auto-configuration (SLAAC), which allows devices to generate their IPv6 addresses without requiring manual configuration or the use of DHCP (Dynamic Host Configuration Protocol) servers. SLAAC enables devices to configure addresses dynamically, reducing the administrative overhead associated with IP address allocation. Additionally, IPv6 includes built-in support for features like neighbour discovery, which simplifies the process of locating and communicating with devices on the same network.
  1. Enhanced Security: IPv6 includes built-in security features that offer improved network security compared to IPv4. One such feature is IPsec (Internet Protocol Security), which provides encryption, authentication, and integrity verification for IPv6 packets. IPsec helps protect the confidentiality and integrity of data transmitted over IPv6 networks. With IPv4, IPsec was an optional feature that had to be implemented separately, whereas in IPv6, it is an integral part of the protocol.
  1. Efficient Routing and Network Management: IPv6 incorporates hierarchical addressing and improved routing protocols, leading to more efficient routing and network management. With IPv6, the hierarchical addressing structure allows for better aggregation of addresses, resulting in smaller routing tables and more efficient routing decisions by routers. Additionally, IPv6 includes protocols like ICMPv6 (Internet Control Message Protocol version 6), which provides better network management capabilities, including diagnostic and troubleshooting functionalities specific to IPv6 networks.

When placing services on available infrastructure, several key issues should be taken into consideration:

  1. Fault Tolerance: Building a resilient system is essential to handle failures gracefully. Infrastructure should be designed to handle hardware failures, network issues, or software crashes. Redundancy and fault-tolerant mechanisms, such as replication or clustering, should be implemented to ensure high availability and minimise downtime.
  1. Scalability: Considering the potential growth and changing demands of the services is crucial. The infrastructure should be scalable to accommodate increasing loads. Horizontal scaling (adding more machines/nodes) or vertical scaling (upgrading resources within a machine/node) should be planned to handle increased demand.
  1. Performance and Latency: Minimising latency and optimising performance are critical factors in providing a good user experience. Services should be placed in close proximity to their users to reduce network latency and improve response times.

Horizontal partitioning involves dividing the data or workload of a service across multiple independent instances or databases, each responsible for handling a subset of the overall workload. This approach helps distribute the load evenly and allows for better scalability and performance. Here's how it works:

  1. Data Partitioning: Analyse the data used by the service and identify key criteria for partitioning, such as customer ID, geographical region, or time range. Split the data into logical partitions based on these criteria.
  1. Partition Distribution: Distribute the partitions across multiple instances or databases. Each instance is responsible for storing and processing a specific subset of the data. This distribution can be based on a consistent hashing algorithm or predefined mapping rules.
  1. Load Balancing: Implement a load balancing mechanism to route incoming requests to the appropriate instances based on the partitioning scheme. This ensures that the workload is evenly distributed across the instances and prevents any single instance from becoming a bottleneck.
  1. Elastic Scaling: Monitor the workload and scale the number of instances dynamically based on the demand. As the demand increases, new instances can be added to the pool to handle additional partitions and distribute the load. Similarly, instances can be removed during periods of low demand to optimise resource usage.
  1. Data Consistency: Implement mechanisms to ensure data consistency across partitions. Depending on the application's requirements, eventual consistency or synchronisation techniques like distributed transactions or conflict resolution protocols may be employed.

Horizontal partitioning allows for better scalability as the workload can be distributed across multiple instances, enabling parallel processing and reducing contention. It also provides improved fault tolerance, as the failure of one instance affects only a subset of the data, minimising the impact on the overall service. Additionally, by distributing the data geographically or based on other criteria, services can be closer to users, reducing latency and improving performance.

Overall, horizontal partitioning or sharding is a powerful strategy to manage services with high demand by distributing the workload and data across multiple instances, enabling better scalability, fault tolerance, and performance.

(c)

The two-phase commit (2PC) protocol is not robust to the situation where a participant crashes after it has voted "Yes." The 2PC protocol is a distributed protocol used to achieve atomicity in a distributed transaction involving a coordinator and multiple participants. It ensures that all participants either commit or abort the transaction consistently. The protocol proceeds in two phases:

  1. Prepare Phase:
    • The coordinator sends a "Prepare" message to all participants, asking them to vote on whether they can commit the transaction.
    • Each participant, upon receiving the "Prepare" message, internally determines if it is ready to commit and votes accordingly. If everything is fine at the participant's end, it votes "Yes" and prepares to commit.
  1. Commit Phase:
    • The coordinator collects votes from all participants. If all participants vote "Yes," the coordinator sends a "Commit" message to all participants, instructing them to commit the transaction.
    • Participants, upon receiving the "Commit" message, perform the actual commit operation and acknowledge the coordinator once they have successfully committed.

The issue arises when a participant crashes after it has voted "Yes" during the Prepare Phase. In this situation:

  1. If the participant crashes before receiving the "Commit" message:
    • The coordinator may assume that the participant has voted "Yes" and proceeds with sending the "Commit" message to other participants.
    • The crashed participant will not receive the "Commit" message and will not commit the transaction. This results in an inconsistency where some participants have committed while others have not.
  1. If the participant crashes after receiving the "Commit" message:
    • The participant, upon recovery, may not be aware of the "Commit" message and may not commit the transaction, assuming it was aborted.
    • This again leads to an inconsistency where some participants have committed while others have not.

In both cases, the protocol fails to achieve atomicity and consistency in the presence of participant failures. The 2PC protocol assumes that participants remain operational throughout the transaction, and any participant failure during the process can lead to inconsistencies.

To address this limitation, more advanced protocols like the Three-Phase Commit (3PC) have been developed, which provide improved fault tolerance and can handle participant failures in a more robust manner.

The leader election problem refers to the task of selecting a single process or node as the leader among a group of distributed processes or nodes in a distributed system. The leader is responsible for coordinating and making decisions on behalf of the group. The leader election algorithm ensures that only one process becomes the leader and that the selection process is fair and efficient.

(e)

One of the well-known leader election algorithms that tolerates failures in a distributed system is the Bully algorithm. The Bully algorithm assumes that every process in the system has a unique identifier (ID) and that processes can communicate with each other through message passing.

Assumptions:

  1. Each process has a unique identifier (ID).
  1. Processes can communicate with each other through message passing.
  1. Processes are aware of the total number of processes in the system.

Outline of the Bully algorithm:

  1. Initialisation:
    • Each process P broadcasts an "Election" message to all processes with higher IDs.
    • If a process P receives an "Election" message, it compares its own ID with the sender's ID. If the sender has a higher ID, P replies with a "OK" message. Otherwise, it ignores the message.
  1. Election Phase:
    • If a process P does not receive any "OK" message after sending "Election" messages to processes with higher IDs, it declares itself as the leader.
    • If a process P receives an "OK" message, it waits for a predefined period for the "Coordinator" message. If it receives a "Coordinator" message, it acknowledges it and joins the network as a non-leader process. Otherwise, it continues with the election process.
  1. Coordinator Phase:
    • If a process P receives a "Coordinator" message, it acknowledges it and joins the network as a non-leader process.
    • If a process P does not receive any "Coordinator" message, it assumes that it is the leader and broadcasts a "Coordinator" message to all other processes.

Properties achieved by the Bully algorithm:

  1. Completeness: Eventually, a leader is elected, even if some processes fail to respond during the election process.
  1. Validity: The elected leader is always a process with the highest ID among the participating processes.
  1. Fault-tolerance: The algorithm tolerates failures by ensuring that the highest-ID process that is alive becomes the leader. If a higher-ID process fails to respond, the lower-ID process will become the leader.
  1. Efficiency: The Bully algorithm has a time complexity of O(n^2) in the worst-case scenario, where n is the number of processes. However, it can be optimised to reduce the number of messages exchanged by using techniques like timeouts and heartbeat messages.

(a)

The CAP (Consistency, Availability, Partition tolerance) Theorem is a fundamental concept in distributed systems that states that it is impossible for a distributed data system to simultaneously provide all three of these properties: Consistency, Availability, and Partition tolerance.

  1. Consistency (C): Consistency refers to the property where all nodes in a distributed system see the same data at the same time. In a consistent system, when a write operation is performed, all subsequent read operations should return the updated value. Maintaining strong consistency requires synchronous communication and can lead to increased latency.
  1. Availability (A): Availability refers to the property that every request to a non-failing node in the distributed system receives a response, even in the presence of failures. An available system remains operational and responsive despite individual node failures. Availability can be achieved by allowing eventual consistency or relaxing the consistency guarantees.
  1. Partition tolerance (P): Partition tolerance refers to the system's ability to continue functioning and providing consistent and available services even when network partitions occur. Network partitions can result in nodes being unable to communicate with each other, leading to a split in the system.

Differences between AP and CP systems:

  1. AP systems: AP systems prioritize Availability and Partition tolerance over strong Consistency. In AP systems, when a network partition occurs, the system remains available by allowing each partition to continue functioning independently. As a result, the system may experience eventual consistency, where different partitions may have inconsistent data for a certain period. Examples of AP systems include Amazon DynamoDB and Cassandra.
  1. CP systems: CP systems prioritize Consistency and Partition tolerance over Availability. In CP systems, strong Consistency is maintained even in the presence of network partitions, ensuring that all nodes have the same view of the data. To ensure consistency, CP systems may sacrifice availability by making certain nodes unavailable during partitions until consistency is restored. Examples of CP systems include Google Spanner and MongoDB with strict consistency settings.

(b)

An example of an AP system is Apache Cassandra. Cassandra is a distributed NoSQL database designed to provide high availability and fault tolerance while sacrificing strong consistency. It follows a decentralised peer-to-peer architecture with no single point of failure

  1. Data Replication: Cassandra replicates data across multiple nodes using a technique called consistent hashing. Each data item is assigned to multiple nodes in the cluster using a partitioning algorithm. This replication provides fault tolerance and allows the system to continue operating even if some nodes fail.
  1. Eventually Consistent: Cassandra provides eventual consistency, meaning that updates to data will propagate across replicas asynchronously. During this propagation, different replicas may have different versions of the data, and it takes time for all replicas to converge to the same state. This approach allows Cassandra to prioritise high availability and partition tolerance while sacrificing strict consistency.
  1. Read and Write Operations: Cassandra allows clients to perform read and write operations on any replica. Clients can specify the desired consistency level for each operation, ranging from strong consistency to eventual consistency. When a client reads data with eventual consistency, it may receive different versions of the data depending on which replica it contacts.
  1. Conflict Resolution: In cases where multiple replicas receive conflicting updates during network partitions, Cassandra uses a mechanism called hinted handoff and a process called read repair. Hinted handoff ensures that writes can still be accepted by replicas even during partitions, and read repair triggers a background process to reconcile any inconsistencies when data is accessed.

Why Apache Cassandra is considered an AP system:

It ensures high availability by allowing read and write operations to be performed on any replica and by providing fault tolerance through data replication across multiple nodes. However, due to eventual consistency, different replicas may have different versions of data for a certain period, and strict consistency guarantees are sacrificed. This trade-off enables Cassandra to provide high availability and fault tolerance in distributed environments, making it well-suited for scenarios where scalability, availability, and partition tolerance are essential, such as large-scale web applications and systems handling big data.

Network eavesdropping and man-in-the-middle attacks. Secure channels play a crucial role in addressing these threats by providing confidentiality, integrity, and authentication.

  1. Network eavesdropping: In a distributed system, communication occurs over a network, making it susceptible to eavesdropping. Network eavesdropping refers to an unauthorised individual intercepting and monitoring the communication between different components or nodes in a distributed system. The eavesdropper can gain access to sensitive information, such as passwords, personal data, or confidential business information. This threat compromises the confidentiality of the system.

Secure channels address network eavesdropping by employing encryption techniques. When communication is established over a secure channel, the data is encrypted before transmission, ensuring that even if an attacker intercepts the data, they cannot understand its contents without the decryption key. Encryption ensures confidentiality and prevents unauthorised access to sensitive information.

  1. Man-in-the-middle (MITM) attacks: A man-in-the-middle attack occurs when an attacker intercepts communication between two parties and impersonates both ends, making them believe they are directly communicating with each other. The attacker can intercept, alter, or inject malicious content into the communication flow, compromising the integrity and authenticity of the data being transmitted. MITM attacks pose a significant threat to distributed systems, as they allow attackers to manipulate and control the communication between components.

Secure channels mitigate MITM attacks through various mechanisms, including mutual authentication and message integrity checks. Mutual authentication ensures that both communicating parties can verify each other's identities before establishing a connection. This prevents an attacker from impersonating either party. Additionally, secure channels can use message authentication codes (MACs) or digital signatures to ensure the integrity of the transmitted data. These techniques allow the receiving party to verify that the received data has not been tampered with during transmission.

A distributed Message Queue system is a fundamental component in distributed computing environments for enabling asynchronous communication and decoupling the sender and receiver of messages. It operates based on the following principles:

  1. Producers and Consumers: The Message Queue system facilitates communication between two entities: producers and consumers. Producers generate and send messages, while consumers receive and process them. Producers and consumers can be located on different nodes or systems within a distributed network.
  1. Message Queues: The Message Queue system acts as an intermediary, providing a storage mechanism called message queues. Message queues act as buffers, allowing producers to send messages without requiring immediate processing by the consumers. The messages remain in the queue until the consumers retrieve and process them.
  1. Asynchronous Communication: The distributed Message Queue system enables asynchronous communication between producers and consumers. Producers send messages to the message queues and continue their operations without waiting for immediate responses from consumers. This decoupling of communication enables scalability, fault tolerance, and flexibility in distributed systems.
  1. Reliable Delivery: The Message Queue system ensures reliable message delivery. Once a message is placed in a queue, it remains there until it is consumed or expires. The system guarantees that messages are not lost, even if the producer or consumer nodes experience failures or downtime. The reliable delivery mechanism typically includes acknowledgment mechanisms, retries, and persistence to ensure message durability.
  1. Message Routing: Distributed Message Queue systems support various message routing strategies. Messages can be sent to specific queues, allowing targeted communication between producers and consumers. Additionally, message headers or properties can be used to define routing rules, enabling flexible and dynamic message routing based on specific criteria

The Ricart-Agrawala mutual exclusion algorithm can be extended to solve the local mutual exclusion problem in a network of processes. The local mutual exclusion problem ensures that no two neighboring processes can execute a critical action simultaneously. Here is an algorithm that achieves this:

  1. Initialization:
    • Each process maintains a list of its neighbors.
    • Each process maintains a queue of pending requests for accessing the critical section.
    • Each process maintains a local variable, "reply," initialized to false.
  1. Requesting Critical Section:
    • When a process wants to enter the critical section, it sends a request message to all its neighbors.
    • The process sets its "reply" variable to false to indicate that it is awaiting permission.
  1. Receiving Request Messages:
    • When a process receives a request message from a neighbor, it checks its own state:
      • If it is currently in the critical section or has already sent a request message to another neighbor, it adds the request to its pending queue.
      • Otherwise, if it is not in the critical section and has not sent a request message, it sends a reply message to the requesting neighbor.
  1. Receiving Reply Messages:
    • When a process receives a reply message from a neighbor, it marks the corresponding neighbor as having replied.
    • If all neighbors have replied, and the process is at the front of its pending queue, it enters the critical section.
  1. Exiting Critical Section:
    • When a process finishes executing the critical section, it removes its request from its pending queue.
    • If the pending queue is not empty, it sends a request message to the next neighbor in the queue.

By using the Ricart-Agrawala algorithm and extending it with the concept of neighbors, we ensure that no two neighboring processes enter the critical section simultaneously. The algorithm achieves this by maintaining a queue of pending requests, allowing only one process at a time to enter the critical section, and ensuring that replies are received from all neighbors before entering the critical section.

Important Points!

  • A hub is a simple device that receives incoming data packets and broadcasts them to all connected devices in the network, regardless of their destination. It operates at the physical layer of the network and does not perform any processing or filtering of the data.
  • On the other hand, a switch is a more intelligent device that examines the destination address of each incoming data packet and forwards it only to the appropriate connected device.
