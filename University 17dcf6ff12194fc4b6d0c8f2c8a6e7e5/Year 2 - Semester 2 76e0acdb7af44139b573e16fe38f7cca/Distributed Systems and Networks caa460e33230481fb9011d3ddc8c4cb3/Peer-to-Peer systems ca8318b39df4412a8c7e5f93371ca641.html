Napster

  • First working, large-scale application of p2p paradigm (1999)
  • Famous for music exchange in MP3 format
  • Up to millions of users simultaneously downloading/uploading music files
  • Shut down for copyright infringement in 2001

Lessons Learned

  • Large-scale p2p systems are technically feasible
    • Although not a pure p2p system
  • Allocation of ‚Äúservers‚Äù to ‚Äúclients‚Äù based on network locality can reduce network load
    • List of peers sorted by number-of-hops
  • Limitations of server-based indexing, it is likely to become a bottleneck
    • File location requests and index update operations can overload centralised servers
  • Specific features of shared files can simplify the design
    • Music files are never updated, hence ensuring replica consistency is simple

BitTorrent

  • Well know p2p system for file sharing
  • Enable sharing large files (e.g., movies)

Reducing BitTorrent‚Äôs Centralisation Problem?

BitTorrent evolved to reduce its centralisation problems:

Every peer acts like a tracker server

Peers are organised in a Distributed Hash Table (DHT), i.e., a ring overlay network

  • Each DHT node has a unique ID, randomly chosen
  • The ID defines the virtual position of a node within the DHT
  • The distance between two IDs reflects the virtual distance between corresponding nodes in the DHT
  • Each DHT node knows many neighbour (close) nodes and a few further nodes
  • A specific node can be found starting from any node in a number of steps logarithmic in the size of the DHT
  • Easy to maintain the overlay network as nodes join and leave
  • To find a file, its hash is used to find the node in the DHT that acts as tracker for that file
  • The tracker is decentralised
  • Magnet links defines file hash
    • They replace .torrent files
