Fundamental Models

All distributed systems share a number of properties:

  • Processes communicate over a network
    • Interactions take time and can affect performance
  • Processes, machines and network infrastructure can fail
    • Need to handle failures to ensure system reliability
  • Systems widely distributed over distant machines and Internet-wide networks
    • Larger cyberattack surface, need to ensure system cybersecurity

All these models below help to capture these properties!

Interaction Model

Conventional algorithms

  • Behaviour and state determined by sequential steps of the single executing process

Distributed algorithms

  • Behaviour and state are determined by the intertwining of the sequential steps of each involved process
  • A distributed algorithm includes the transmission of messages
  • Processes proceed at different rates and send/receive messages at different rates
  • The system state is distributed as well, no global notion of system state
  • Global system state: combination of the local state of all involved processes

How can I ensure that all three processes always share the same global state?

  • Global state = [State 1, State 2, State 3]

Performance of communication channels

  • Latency: from start of transmission to start of reception, measured in seconds
    • Network latency
    • Network card delay
    • OS delay
  • Bandwidth: total amount of information that can be transmitted over it in a given time
    • Measured in bit per second
  • Jitter: variation in the time it takes to deliver a series of messages
    • Measured in units of time (e.g., ms)
    • e.g., media stream

Computer clocks and timing events

  • If two processes running on different physical machines read their clocks at the same time, they would get different time values
  • Even if the clocks are set to the same time initially, their value will eventually vary quite significantly unless corrections are applied
    • Drift: clocks update their counting register with different frequencies, hence they deviate over time
    • Drift rate: time difference for time unit with respect to an ideal clock
      • e.g., High-precision quartz clock drift rate is 10-7-10-8 secs/sec
  • Corrections can be applied through synchronisation protocols, but the accuracy is affected by network delays
  • No global notion of time, cannot rely on local clocks for synchronisation purposes!!!

Synchronous vs Asynchronous Systems

Difficult to set limits (i.e., make assumptions) on:

  • Process execution time
    • Shared resources within the same machine (i.e., CPU, memory, disk), so different processes compete to access these resources and may need to wait
  • Message latency
    • Network bandwidth is limited
  • Clock drift rate

Estimations are possible but replying on them is not safe

(e.g., on average, each process completes its local task in 0.1 seconds; if we rely on the assumption that each process completes its local task in 0.2 seconds, what happens when it takes longer?)

Synchronous Distributed Systems

  • Assumptions: known upper and lower bounds on processing time, network latency and clock drift rate
  • Simpler design
  • Useful when starting the design of a new distributed algorithm, to draft a preliminary version and validate it
    • (e.g., can use timeouts to detect failures)

Asynchronous Distributed Systems

  • Assumptions: no bound on process execution speed, message transmission time and clock drift rate
  • Harder to design
  • Models the Internet and real systems much better than synchronous assumptions
  • Any solution that is valid for an asynchronous distributed system is also valid for a synchronous one

Failure Model

  • Processes and communication channels fail when they depart from the expected (i.e., correct/desirable) behaviour

The failure model defines the possible ways a process or communication channel can fail:

  • Process/Channel Omission failures
  • Arbitrary failures
  • Timing failures

Process Omission Failures

Crash: process has halted and will not execute any further steps of its program ever

Can other processes detect that a process has crashed?

  • Synchronous model
    • Can use timeout: if a process P does not reply to a message within a certain timeout, then P is considered crashed
    • All the processes can detect a crashed process: fail-stop failure
  • Asynchronous model
    • Maybe the process is not replying because it is slow or the message is not arrived yet
    • Other process cannot reliably detect a crashed process: crash failure

Channel Omission Failures

  • Send omission failure
  • Receive omission failure

Arbitrary (Byzantine) failures

  • Send or transmit arbitrary messages at arbitrary times
  • Commit omissions
  • A process may stop or take an incorrect step
  • A malicious process (e.g., infected by malware)

Timing Failures

  • Relevant in synchronous systems
  • Time limits on process execution time, message delivery time or clock drift rate are violated

Dealing with Failure

Masking

  • Process replication to tolerate process omission failures
  • Retransmission to tolerate channel omission failures

Conversion

  • Use checksum to detect corrupted messages and discard them; arbitrary failure converted to channel omission failure

Reliability of one-to-one communication

  • Validity: any message in the outgoing message buffer is eventually delivered to the incoming message buffer
  • Integrity: the message received is identical to one sent, and no messages are delivered twice

Guaranteed by System (important!)

  • Timeout-based retransmission can guarantee validity
  • Checksum + retransmission can guarantee “part of” integrity (i.e., “the message received is identical to one sent”)
  • Discarding duplicated received messages is required to guarantee “the other part of” integrity (i.e., “no messages are delivered twice”)

Security Model

GOALS:

  • Protecting objects (e.g., access control)
  • Securing processes and their interactions

The enemy (Threat Model)

  • Threats to processes: identity spoofing, malicious code
  • Threats to communication channels: copy/alter/inject messages
  • Denial of service (DDoS)

Cryptography

  • Encrypt exchanged and stored data to preserve confidentiality

Authentication

  • Based on cryptography and message digests; digital signatures
  • Prove the identity of sender and receiver

Secure Channels

  • Authenticated parties
  • Provide integrity and privacy of transmitted data
