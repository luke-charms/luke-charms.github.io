Distributed Transactions

Transaction:

A transaction is a set of related sequential operations that is guaranteed by the server to be atomic in the presence of multiple clients and server crashes
  • Clients may need a sequence of requests to be atomic
    • Free from interference by operations requested by other concurrent clients
    • Either all operations must be successfully completed or none must complete
  • All objects managed by a server remain in a consistent state
  • Transactions deal with:
    • Crash failures of processes and
    • Omission failures in communication

(but not any type of arbitrary (or Byzantine) behaviour)

Example:

// Transaction T:
balance = b.getBalance();
b.setBalance(balance + 75);
a.withdraw(75);

ACID Properties

Properties of database transactions intended to guarantee data validity:

  • Atomicity - a transaction must be all or nothing
  • Consistency - a transaction takes the system from one consistent state to another consistent state
  • Isolation - each transaction must be performed without interference from other transactions
  • Durability - After a transaction has completed successfully, all its effects are saved in permanent storage

Transactions – Crash Recovery

Objects need to be recoverable to support transactions

In the event of a server crash, the changes made by completed transactions must be durable and stored in permanent storage

When a new process replaces a crashed server, it can recover the objects to ensure the all-or-nothing effect of transactions

Uncommitted transactions are aborted, and recovery procedures are performed to restore objects to their last committed values.

Transactions - Concurrency

Synchronization of operations in different transactions is necessary to ensure isolation.

  • Just executing transactions serially wouldn’t be acceptable because of low resource utilisation and long waiting time
  • Concurrent execution of transactions is allowed if it produces the same effect as a serial execution.

that is, if they are serially equivalent or serializable

  • Inconsistent retrievals

Concurrency Controls

Serial Equivalence is used to inform concurrency control protocols

Three approaches are commonly used:

  1. Locks
  1. Optimistic Concurrency Control
  1. Timestamp ordering

Locking

Locks are used to regulate access and ensure transaction isolation and data consistency

Lock (labelled with the transaction identifier) set on each object just before it is accessed

While an object is locked, only the transaction that it is locked for can access that object

Lock removed when the transaction has completed

Can lead to deadlocks, with transactions waiting for each other to release locks

Effective locking controls maintain consistency but excessive locks can lead to contention and performance issues

Optimistic Concurrency Control

Assumes that conflicts between concurrent transactions are infrequent

Read Phase:

  • Transactions read required data items from the application.
  • Each data item is associated with a timestamp or version number.

Validation Phase:

  • Transactions recheck the data items to detect conflicts.
  • Comparisons are made between recorded timestamps/versions and current ones.
  • If all timestamps/versions match, the transaction proceeds; otherwise, it is rolled back.

Write Phase:

  • If the validation phase succeeds, transactions write changes and commit.
  • Data items receive new timestamps/versions to reflect the updates.

Timestamp Ordering

Total order among transactions based on their start timestamp:

  1. Timestamp assignment: When a transaction enters the system, it is assigned a unique timestamp that reflects its starting time.
  1. Read and write operations: When a transaction reads an object, it compares its timestamp with the timestamp of the last writing transaction for that object. If the reading transaction's timestamp is earlier, it implies that it is trying to read an outdated value.
  1. Validation phase: The system checks if the transaction's read and write operations conflict with other transactions that have executed concurrently.

    (Conflicts occur when multiple transactions access the same data item, and at least one of them is a write operation)

  1. Commit or Abort

Recoverability from Aborts

Successful committed transactions' changes are recorded and stored.

Changes made by aborted transactions are ignored and not stored.

This prevents the impact of aborted transactions on concurrent transactions.

The server maintains the integrity of the overall system by ensuring aborted transactions do not affect other transactions.

Dirty Read

A dirty read is a phenomenon that can occur one transaction reads data that has been modified by another transaction but has not yet been committed

When a transaction reads uncommitted or "dirty" data

This can be problematic because a transaction may use incorrect data, and make decisions or take actions based on unreliable information causing inconsistent or invalid actions.

  • Recoverability of transactions relies on a strategy that postpones the commit of a transaction until after the commitment of any other transactions that have observed its uncommitted state.
    • Transaction U delays its own commit until transaction T has successfully committed.
    • If transaction T aborts, transaction U must also abort.
  • Cascading aborts occur when the effects of an aborted transaction ripple through other transactions. To prevent cascading aborts:
    • Any transactions that have seen the effects of transaction U must be aborted if transaction T aborts
    • Transactions are only permitted to read objects that have been written by committed transactions.

Premature Writes

A premature write is when multiple transactions attempt to write to the same object or data item

One of the transactions successfully completes the write operation, modifying the state of the object

However, this particular transaction encounters an issue or conflict, leading an abortion or rollback

The aborted transaction causes the object to be in an inconsistent state or a state that does not align with the intended or expected outcome

Aborts can be implemented by restoring the before image of the writes

  • a =£100 is the before image of T’s write
  • a =£105 is the before image of U’s write
  • If T commits and U aborts, the balance is restored to £105
  • If T aborts and U commits, the balance is restored to £100 – should be £110
  • If T aborts and U aborts, the balance is restored to £105 – should be £100

When a database uses before images, write operations must be delayed until earlier transactions that updated the same objects have either committed or aborted

Coordinator

A distributed transaction refers to a transaction that interacts with objects handled by multiple servers.

Atomicity is a crucial property in distributed transactions, where the transaction is either committed by all participating servers or aborted by all of them.

  • One of the servers assumes the role of a coordinator, responsible for ensuring that all servers reach the same outcome regarding the transaction.
  • The coordinator coordinates the transaction's execution across servers to guarantee consistency and maintain atomicity.

Two-Phase Commit Protocol

The coordinator writes its decision to disk for durability.

If the coordinator crashes after the prepare phase but before broadcasting the decision, other nodes are unaware of its decision.

Replicas participating in the transaction cannot commit or abort after responding with "OK" to the prepare request to maintain atomicity

The algorithm is blocked until the coordinator recovers and broadcasts its decision.
