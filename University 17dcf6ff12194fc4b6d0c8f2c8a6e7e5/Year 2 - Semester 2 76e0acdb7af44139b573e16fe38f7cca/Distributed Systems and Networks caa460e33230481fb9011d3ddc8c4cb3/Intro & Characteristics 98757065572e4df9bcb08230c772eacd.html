New requirements for Distributed Systems

Decentralised Networks

- Centralisation can lead to

  • Performance bottleneck
  • Single-point-of-failure
  • Censorship

- Decentralised Distributed Systems as a way to remove centre-based architectures

  • Peer-to-peer network
  • Blockchain

Trend towards decentralised systems is a move towards removing centralised components of the system. Usually, DS are built up of components that all do one thing and do it well, but decentralised systems rework this.

Internet of Things

  • Pervasive network of sensors and actuators
    • Smart home
    • Smart building
    • Smart vehicles
    • Industry 4.0
  • IoT devices are not general purpose and have limited resources
    • Need to (aggregate and) send data to more powerful devices for analysis
    • Usually, data sent to the cloud

Edge Computing and Fog Computing

Edge computing:

- More powerful devices to be deployed at the edge of the network

- Edge devices can carry out (part of) the computation where data is generated

Fog computing:

- Even more resourceful devices can be placed between the edge and the cloud

- Fog devices can carry out (part of) the computation very close to where data is generated

Heterogeneity

Variety and difference of networks, hardware, operating systems, programming languages, implementations

  • E.g., Internet protocols to mask network heterogeneity
  • E.g., integer values can be represented in different ways on different HW
  • E.g., different OSs provide different interfaces to access/use network protocols
  • E.g., different programming languages use different data representations
  • E.g., software written by different developers cannot communicate among each other, unless standards have been agreed and adopted

Middleware: software layer that provides a programming abstraction as well as masking the heterogeneity of the underlying networks, HW, OSs and programming languages

Openness

An open system can be extended and reimplemented in various ways

  • To add more computational/storage resources
  • To integrate additional services

The more easily a system can be extended, the more open it is

It requires key interfaces to be published

  • Similar to standardisation but less formal and (thus) faster

Scalability

A system is scalable if it remains effective when there is a significant increase in

  • The number of resources, and/or
  • The number of users
  • Ability to dynamically scale - scale out when workload increases and scale back in when workload decreases

Controlling the cost of physical resources

  • Ease to add resources (e.g. machines) to meet growing demand

Controlling the performance loss

  • Algorithms performance should remain acceptable as problem size grows

Preventing software resources running out

  • Set/reserve the proper size for core data structure

Avoiding performance bottlenecks

  • Avoid centralisation

Failure handling

Detect failures

  • Accuracy issues, e.g. server failure or network failure?

Mask failures

  • Message retransmission
  • Re-route to a replica

Tolerate failure

  • Inform the user

Recovery from failure

  • Data can be rolled back to a consistent state after a crash
  • Service availability can be restored after a crash

Concurrency

More clients can try to access the same resource at the same time

  • How to handle that?

Solution 1: serialise access

  • Safe, but limits throughput

Solution 2: allow for concurrent accesses

  • Maximise throughput, can lead to inconsistencies

Need to encapsulate the resource and regulate accesses based on application specific requirements
