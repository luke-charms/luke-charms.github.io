What is a Web Service?

A web service provides an interface that clients can access over the Internet

From web servers to web services

  • Web servers: general-purpose browsers interact with web servers over the Internet via HTTP
    • The interface provided by the web server is very generic, and this restricts the scope of applications
  • Client-server model: client and server are application-specific and the particular application logic dictates the interface used for communication
  • Web services: application-specific client interacts with a remote service via a functionally specialised interface over the Internet

Web servers: a general-purpose browser as client to retrieve resources via HTTP

Web services: application-specific client-server paradigm over the Internet

Communication paradigms

  • Asynchronous communication
    • For time-consuming operations, replies can be received later on
    • E.g. processing of a booking
  • Synchronous communication
    • For interactive, fast operations, i.e. request-reply
    • E.g. booking processing status enquires - doesn’t require much computation

Loose coupling

  • Goal: minimise dependencies between different web services
    • To avoid that changing one web service requires the update of other services
  • How to achieve loose coupling:
    • Focusing on interfaces rather than implementation
      • So that the implementation can be changed without affecting the interface
    • Designing simple and generic interfaces
      • Minimal, basic interfaces are less likely to require change over time
    • Opting for asynchronous rather than synchronous communication
      • Looser coupling and time

SOAP Protocol

Simple Object Access Protocol (2)

  • Synchronous/Asynchronous interactions over the Internet
  • Defines how to use XML to represent the content of messages
  • Defines how messages are exchanged
    • HTTP, SMTP, TCP, UDP,…
  • It is a WWW recommendation!
  • Specifies how to combine XML and HTTP to enable client-server communication over the Internet

RESTful Web Services

  • With SOAP, there is a significant coupling between clients and web services
    • Any change to the API requires all the clients to be updated accordingly
  • An alternative, simpler approach is Representational State Transfer (REST)
    • Uses a simple minimum uniform set of operations
      • HTTP methods: GET, PUT, POST, DELETE
    • Focuses on resources rather than the operations allowed on them
      • Each resource can be accessed through a specific, unique URL
    • Allows for looser coupling compared to SOAP because it removes the dependency on application-specific APIs
      • The emphasis is on the manipulation of data resources rather than on interfaces

Use HTTP methods explicitly

  • To create a resource on the server, use POST
  • To retrieve a resource, use GET
  • To change the state of a resource or to update it, use PUT
  • To remove or delete a resource, use DELETE

Be stateless

- Allows for better scalability by requiring clients to send complete and independent requests

- No concept of sessions spanning more than one request

Expose directory structure-like URIs

Each resource is accessible via a URI

The structure of a URI should be straightforward, predictable, and easily understood

Directory structure-like URIs

  • Hierarchical
  • Rooted at a single path
  • Branches of the root are subpaths that expose the service's main areas

Examples:

  • http://www.myservice.org/discussion/topics/{topic}
  • http://www.myservice.org/discussion/2008/12/10/{topic}

Transfer XML, JSON or both

Resources need to be represented in some format in order to be transferred

  • Human readable
    • e.g. XML, JSON
  • Also relationships between resources need to be represented
