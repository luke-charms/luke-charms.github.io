Critical Section:

  • When more than one process can access the same code segment
  • Contain shared variables or resources
  • Required to maintain the consistency of the variables
  • The values depend on the sequence of execution of instructions - Race condition

Single Computer System vs Distributed Systems

Single Computer System

  • Memory and other resources are shared between different processes
  • Status of shared resources are easily available

Distributed System

  • No shared memory or common physical clock
  • Do not have complete information of state of system
  • Message passing can be used

Mutual Exclusion Requirements

No Deadlock

  • Two or more sites should not endlessly wait for any message that will never arrive

No Starvation

  • Every site that wants to execute critical section should get an opportunity to execute
  • Every site should not wait indefinitely

Fairness

  • Every site should get a fair chance to execute critical section
  • Execute requests based on the order that they are made

Fault Tolerance

  • Ability to recognise failures to avoid disruption

Safety & Liveness Properties

  • Safety properties so ‘bad things don’t happen’
  • Liveness properties so ‘good things do happen’

Safety properties

  • One process can remain in its critical section at any time (mutual exclusion)
  • Freedom from deadlock, at least one process must be eligible to take an action and enter its critical section

Liveness properties

  • Every process trying to enter its critical section must eventually succeed (livelock or starvation)

Fairness and Performance Properties

Fairness properties

  • Access to critical section should happened in fair order
  • Happened-before ordering
    • if one request to enter the critical section happened before another then it should enter the critical section first

Performance properties

  • Minimise number of messages in sent to each entry/exit operation
  • Minimise client delay when entering/exiting critical section
  • Minimise synchronisation delay which is the time between exiting critical section and the next entering

Mutual Exclusion Distributed Solutions

Three types exist:

Token Based Algorithms

  • Centralised, Ring and Raymond’s algorithm

Non-Token Based Algorithms

  • Lamport’s Algorithms and Ricart-Agrawala

Quorum based approach

  • Maekawa’s Algorithm

Token Based Algorithms

  • A unique token is shared among all sites
  • If a site possesses the unique toke it is allow to enter the critical section
  • Uses sequence numbers to order requests
  • Sequence numbers used to identify old and current requests
  • Each token is unique

(example: Centralised Algorithm, Ring Algorithm, Raymond’s Algorithm)

Centralised Algorithm

Steps:

  1. P3 requests token
  1. Token is granted
  1. P2 requests token
  1. P2 added to coordinators queue
  1. P1 requests token
  1. P3 releases token
  1. P0 grants token to P2, removes P2 from queue

Safety & Liveness Properties

Safety properties:

  • Server ensures grant is only sent after a release is received
  • Queue ensures only one process is eligible to enter critical section

Liveness properties

  • Server maintains a queue so all processes will eventually enter critical section in order

Fairness and Performance Properties

Fairness properties

  • Server maintains a queue so all processes will eventually enter critical section in order
  • Can not ensure happened-before ordering if in an asynchronous system

Performance properties

  • Enter overhead: 2 messages, exit overhead: 0 message
  • Entry delay: message round trip
  • Synchronisation delay: message round trip (release and grant)
  • Single point of failure
  • Server can be bottleneck
  • Server must be known (or elected)

Ring Token Algorithm

Steps:

  1. P0 has the token, its queue is empty
  1. P0 releases the token to P1
  1. P1 accesses the critical process the oldest item in the queue
  1. Removes the oldest item in the queue
  1. P1 has the token, its queue is empty
  1. P1 Releases the token to P2
  1. P2 releases the token to P3
  1. P3 access the critical process with the oldest item in the queue

Safety & Liveness Properties

Safety properties

  • There is only one token and the critical section can only be entered when the process holds the token, therefore only one process can enter the critical section at any time
  • Deadlock free because only one token

Liveness properties

  • Every process gets access to the critical section because the token is passed onto the next process after each critical section

Fairness and Performance Properties

Fairness Properties

  • Access to critical section is not fair
  • Order is based on ring and the location of the token

Performance Properties

  • Enter: 0 to N messages, exit: 0 message
  • Delay entering: 0 to N messages, delay exiting: 0
  • Synchronisation delay: 1 to N-1 messages
  • Token circulating consumes bandwidth

Raymond’s Tree Based Algorithm (using tokens)

Each node has only one parent

A child node can only send requests to its parents

Each node maintains a FIFO queue of requests

If any node is forwarding privilege to other node and has non-empty queue, then it forwards a request message

Steps:

  1. If a node i (not holding the token) wants to use the token, so that it can enter the critical section, it sends a request to its parent, node j.
    • If node j FIFO queue is empty, node j shifts i into its FIFO queue; j then issues a request to its parent, k, that it desires the token
    • If node j FIFO queue is not empty, it simply shifts i into the queue
  1. When node k has token and receives the request from j it sends token to j and sets j as its parent
  1. When node j receives the token from k, it forwards the token to i and i is removed from the queue of j
    • If the queue of j is not empty after forwarding the token to i, j must issue a request to i in order to get the token back

Example:

  1. P3 wants the token, and adds itself to its own FIFO queue
  1. Sends a request message to P2
  1. P2 receives the request from P3, and adds P3 to its queue
  1. P2 sends the request to P1
  1. P1 receives the request from P2 and adds P3 to its queue
  1. Passes the request to its parent P0
  1. P2 also wants the token, and adds itself to its queue
  1. P0 receives the request message, and sends the token to P1.
  1. P1 removes the top element and passes the token to P2
  1. P2 passes the token to P3 and removes 3 from its queue, temporary changing its direction so P3 is the root
  1. P3 accesses critical section
  1. When P3 is done it clears its queue and sends the token to P2 and changes the root

Safety & Liveness Properties

Safety properties

  • Only one process can hold the privileged token
  • Deadlock is impossible

Liveness properties

  • Algorithm can cause starvation as it used a greedy strategy, where a site can execute the critical section on receiving the token even when its request is not on the top of the request queue

Fairness and Performance Properties

Fairness properties

  • Everyone is allowed to hold the token, non greedy strategy

Performance Properties

  • Enter: 0 to N messages, exit: 1 message
  • Delay entering: 0 to N messages, delay exiting: 0
  • Synchronisation delay: ( log N )/ 2
  • The message complexity of this algorithm is O(log N) as the average distance between any two nodes in a tree with N nodes is log N

Issues with Token Based Algorithms

How do we deal with lost tokens?

Process failures how do we detect them?

—> Employ simple algorithm at network cost

Non-Token Based Algorithms

  • A site communicates with other sites in order to determine which sites should execute critical section
  • Uses timestamps instead of sequence number to order requests
  • Whenever a request is received it gets a timestamp
  • Timestamps used to resolve any conflicts between requests
  • Maintain a logical clock (logical clock updates according to Lamport’s scheme)

(examples: Lamport’s Algorithm and Ricart-Agrawala)

Ricart-Agrawala Algorithm

  • Terminology

    A site is any computing device which runs the Ricart-Agrawala Algorithm

    The requesting site is the site which is requesting to enter the critical section.

    The receiving site is every other site which is receiving a request from the requesting site.

Steps:

  1. All sites have their own queue
  1. Request messages contain the site id and timestamp - request(id, timestamp)
  1. When a site wants to use the critical section it broadcasts it to all the sites
  1. All sites add it to their queue, and respond iff:
    • the receiving process is not currently interested in the critical section
    • the receiving process has a lower priority based on timestamp
    • Otherwise they will respond to the requesting site
  1. Requesting site waits until all sites have replied to use the critical section
  1. When the requesting site has exited the critical section, it sends any deferred response messages

Safety & Liveness Properties

Safety properties

  • If a process is in the critical section, it doesn’t reply until it exits
  • If more than one process is requesting, process with lowest timestamp will get all N-1 replies first

Liveness properties

  • Total ordering of timestamps ensures each request will eventually get all replies

Fairness and Performance Properties

Fairness Properties

  • Clocks ensue fair ordering

Performance Properties

  • Entry: 2(N-1) messages, Exit: 0 to N-1
  • Entry delay: message round trip, exit delay: 0
  • Synchronization delay: 1 message
  • Better than Lamport’s algorithm

Quorum Based Approach

  • Instead of requesting permission to execute from all possible sites, each site requests only a subset of sites (quorum)
  • Any two subsets of Quorum contains a common site
  • This common site is responsible to ensure mutual exclusion

(example: Maekawa’s Algorithm)

Maekawa’s Algorithm

Divide N processes into N request subsets R

Ri is the request subset for Pi

Maekawa’s algorithm runs within a request subset

Subsets of size K, each process is in M subsets

Maekawa showed that optimal values are: K ~ √N, M=K

Each node has its own:

  • Queue
  • Boolean voted TRUE/FALSE
  • State WANTED/RELEASED/HELD
Initial state: state=RELEASED, voted=FALSE, queue=[]

Pi: wants to enter Critical Section:
Set state=WANTED
multicast request to all processes in Ri
Wait until received K reply messages
Set state=HELD

Pi: wants to exit Critical Section:
Set state=RELEASE
multicast release to all processes in Ri

Pj: receives request from Pi
If	State=HELD or voted=True:    queue request
Else: send reply to Pi and set voted=True

Pj received release from Pi
If queue is empty:    set voted=False
Else:
Px = head of queue
Remove Px  from queue and send Px the reply
Set voted=True

Safety & Liveness Properties

Safety properties

  • Processes can only vote for 1 process at time so only one process can gain enough votes to enter critical section

Liveness properties

  • Very easy to deadlock
  • Two processes can broadcast request at the same time
  • Solved by modifying queue behaviour to use happened-before order

Fairness and Performance Properties

Fairness Properties

  • No, but can be introduced by using Vector clocks to determine the order of requests

Performance Properties

  • Entry: 2√N messages, Exit: √N messages
  • Better than Ricart and Agrawala’s if N > 4
  • Delay entry: 2 messages, delay exit: 1 message
  • Synchronization delay: 2 messages
