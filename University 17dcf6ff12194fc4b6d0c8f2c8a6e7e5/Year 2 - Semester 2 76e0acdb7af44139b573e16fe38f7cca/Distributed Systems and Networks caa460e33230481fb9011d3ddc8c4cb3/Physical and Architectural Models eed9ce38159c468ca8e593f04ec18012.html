Communicating Elements:

  • System-oriented perspective (lower level, i.e., what is running?)
    • Processes communicating via inter-process channels
    • If no process abstraction is available (e.g., sensors), nodes

  • Problem-oriented perspective (higher level, i.e., can I abstract what is running?)
    • Objects (provided interface)
    • Components (both provided and required interfaces)
    • Web services (web-based, thought to be accessed worldwide over the Internet)

Communication Paradigms:

Inter-process communication (network level)

  • Message passing
  • Sockets

Remote invocation (communication protocols based on a two-way exchange)

  • Request-reply protocols (e.g., send a request and receive a response, HTTP)
  • Remote Procedure Call (RPC)
    • Request the execution of a remote function and receive the response
  • Remote method invocation (RMI)
    • Match with problem-oriented communicating entities (e.g., objects)
    • Object-oriented RPC, i.e., invocation of a method of a remote object

Roles and Responsibilities

Client-Server

  • Clients send requests to servers and receive responses back
  • Clients enabled to access shared resources managed by servers
  • Servers can be clients of other servers

Peer-to-peer (p2p)

  • All involved entities play similar roles
    • Same program, same interfaces
    • Running on different machines
  • Clients themselves can become part of the p2p network and provide their resources to
    • Clients are now more resourceful and always-connected
  • Enables decentralised approaches
    • Can improve scalability
    • Workload can be evenly distributed over peers
    • P2p design much more complex than client-server
      • No control over single peers, how to ensure availability?
