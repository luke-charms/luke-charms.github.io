Data Replication and Consistency

Consistency Models

A contract between an application and a data store which says that if the application agrees to obey certain rules, the data store will work correctly

Replication:

Passive:

  • One replica is considered the Primary
  • Clients only communicate with the Primary
  • Any other replicas (backups, Secondarys) are used as back up
  • If Primary fails, backups are ready to take over
  • Leader election is held to elect a new Primary

Active:

  • Client communicates with a group of services (called replicas)
  • When a client writes to a replica; the update is propagated amongst the replicas using total-order multicast

In linearizability; the trace is the most important aspect of the model

Implementation:

Every process must have a perfectly synchronised clock

Message propagation delays must be bounded

Use total-order multicast to ensure replicas handle reads and writes in the same (time-based) order

Sequential Consistency

Makes no statement about timing, but just talks about some sequential program order

(weaker version of Strict Consistency and Linerizability)

Implementation:

  • A read operation made to a replica will return its the local copy
  • A write operation will trigger a Total-Order multicast
  • Replicas respond to multicast by updating local copies and return an acknowledgement
  • All replicas see writes in the same order and reads return values consistent with the writes in the total order
  • No need for synchronised clocks

Causal Consistency

Implementation:

  • Use vector timestamps
  • Write operations are multicasted to all replicas
    • writes are sent with the vector clock
  • On receiving multicast updates, each replica updates its local copy
  • A read uses the local copy
  • No need for synchronised physical clocks
  • Provides more useful guarantees than eventual consistency

Eventual Consistency

Implementation:

  • Update local store and propagate changes in the background
  • To deal with conflict:
    • Reconcile the conflict - this needs a global decision to be made i.e. via consensus
    • OR Roll back
    • OR Delegate to the application
  • Real world example: Amazon Dynamo
    • Shopping cart problem where different versions of the shopping cart can result from concurrent updates and reconciliation needs to occur
    • Result is that items sometimes remain in the basket for a short period of time after they've been deleted

Advantages:

  • Fast reads and writes
  • If there is a split in the network, application can still operate

Disadvantages:

  • Very weak notion of consistency
  • It needs a mechanism to deal with conflict

STRONG eventual consistency

Conflict-Free Replicated Data Type - CRDT

Two types of CRDT:

Operation-based CRDTs

  • On write broadcast operation only
  • On receive broadcast apply operation
  • Often used in collaborative editors

State-based CRDTs

  • On write broadcast full state
  • On receive broadcast merge states
  • For example, Grow-only counter
  • Used in key-value stores, like the Redis data store

Example:

Grow-only Counter that counts events from a set of N processes

The data type is structured in such a way that it automatically resolves any conflicts

Algorithm:

  • Each process has an integer ID: 0..N-1
  • Maintains a vector of counts: count[ 0..N-1 ]
  • Value of counter: sum(count)
  • Increment to counter: count[id] ++
  • Merge two counters: pairwise max of the two vectors
    • i.e. merge(<8, 3, 10>, <7, 6, 11>) = <8, 6, 11>
  • Background replication: process multicasts count vector to other processes

Quorum-based Protocols

  • Replication is intended to improve the reliability of a system
  • How replication is performed can have a big impact on the reliability
    • If there is no fault-tolerance, having multiple replicas makes reliability worse
    • The more replicas you have the greater the probability that any one of them is faulty
  • Using a quorum is a common approach of implementing consistency across a number of replicas in a distributed data store

How?

  • A quorum is a subset of replicas, dynamically selected during operations
  • Clients read from a read quorum and write to a write quorum
  • These two types of quorums intersect to ensure consistency
  • In a system with N replicas, have a quorum of R replicas in the read quorum and W replicas in the write quorum
    • R + W > N (ensures that the read and write quorum must overlap)
    • Typically set R = W = (N+1)/2 to ensure majority

Example:

  • After a write has been made to the quorum, the replicas with the new write can update other replicas in the background
  • If replicas fail or when replicas become available, then quorum reads and writes can continue as long as the remaining available replicas still form a majority
  • For example, replicas can become unavailable due to a network partition
