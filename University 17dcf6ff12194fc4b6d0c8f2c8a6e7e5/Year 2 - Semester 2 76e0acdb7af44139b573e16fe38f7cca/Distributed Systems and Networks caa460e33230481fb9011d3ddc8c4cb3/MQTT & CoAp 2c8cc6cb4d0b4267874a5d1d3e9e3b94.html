MQTT & CoAp

MQTT

Designed to be smaller packet and less overhead than HTTP

Standard design for IoT devices

Messages are published to a broker

Clients subscribe to data streams (e.g., topic temperature)

Topics are hierarchical

i.e., subscribing to house/# includes all of these below:

/house

/house/room/main-light

/house/room/side-light

CoAP (Constrained Application Protocol)

Efficient in low power IoT systems - for longer battery life

RESTful - as familiar as HTTP (requests)

  • Copies code from HTTP (404)
  • Makes interoperability with The Web simple

Low resource devices

  • Low memory footprint
  • Basic packets should fit in an 802.15.4 radio frame

Binary format for protocol (to save space) and typically uses UDP

Features

  • A more modern lightweight design than MQTT
  • DTLS security option
  • Block transfers
  • Resource discovery
  • Push notification
  • Cache model
  • Multicast support

CoAP Header

  • Is VERY small!
  • Code is method (1-10) e.g. GET, or response code (40-255) like HTTP
  • ID helps match messages (response is same)
  • Token is used to match responses
  • Option can mean payload is a URL string
  • Payload can be anything

Dealing with packet loss

CoAP uses confirmable requests

If the client doesn’t receive an ACK back, they will attempt to resend the request

  • Responding with the data on a constrained system can be slow, so the ACK gets sent immediately and the payload later

A message ID is used to keep track of responses

Proxies

Since these devices are limited by nature, proxies can handle requests to them to prevent overwhelming them / requesting when they’re down

In addition, proxies can cache values, allowing nodes to sleep and conserve power

Resource Discovery

Use GET .well-known/core to receive a CoRE description of the resources on the node

This is a standard format for defining resources

This means that IoT objects can be added and their resources discovered automatically
