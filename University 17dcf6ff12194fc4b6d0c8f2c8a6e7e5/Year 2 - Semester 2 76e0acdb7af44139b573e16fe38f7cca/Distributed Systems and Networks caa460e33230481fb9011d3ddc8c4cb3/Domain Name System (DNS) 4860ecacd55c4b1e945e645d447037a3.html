Domain Names

Domain names are delegated from the ICANN through the Top Level Domain (TLD) registrars

  • This allows sites and organisations to be delegated the authority to maintain information about the domains they have registered (e.g., the IP address of their public-facing web server)
  • e.g.,
    • soton.ac.uk through Nominet (.uk)
    • JISC runs the Janet network (.ac.uk)
    • University runs last bit (soton.ac.uk)

Generally operates iteratively

  • Client doing a query initially talks to root name servers
  • Then follows the referrals down the hierarchy to the authoritative name server for a domain

“Zones of authority” are managed by name servers

Looking up data in DNS

  • Most common DNS lookup is host name to IP
  • DNS returns an A record(IPv4) or AAAA record (IPv6) when queried for a specific host/domain name

This implicitly requires that clients (hosts, and indirectly applications) know the IP addresses of a local DNS server (resolver) they can send queries to:

  • Resolvers are local DNS servers that hosts can send queries to. They “resolve” these queries and send them back - Client hosts are configured to use one or more local DNS resolvers
  • May be the ADSL router in a home network
  • Or a DNS server run by iSolutions on the uni campus
  • There are also ‘public’ DNS servers

Examples of DNS record types

DNS lookup

  • Client side is called a DNS resolver
  • Queries are usually UDP packets
  • These are sent to name severs until an answer is found
  • Resolvers and name servers cache results - and can be started with hints
  • Use -dig in terminal:

DNS infrastructure

  • A campus will run its own DNS servers
    • To act as resolvers for internally sourced queries, either for internal names or external names
    • To act as servers for external queries against internal names
  • How do other systems outside the campus know how to find our DNS servers in order to send queries?
    • Specifically, how can an external DNS resolver find the IP of our campus DNS servers?

—> This is where the hierarchy of DNS comes in.

  • ICANN manages a number of “root” DNS servers.
  • The root servers know where the DNS servers for each TLD are.
    • (that is, for .org, .uk, .com, .net etc.)
  • And so on; each server knows where the name servers for their delegated subdomains are

A DNS resolver can therefore be configured with IP addresses for the root servers and then choose one from which to iteratively “walk” the hierarchy from the root in order to resolve a lookup.

Recursive DNS lookup

Requests go to a resolver and it it doesn’t know how to resolve a request it will:

  • First asks a root DNS server (which it already knows the IP of)
  • Second, the root will then tell it to try another DNS server lower down in the hierarchy
  • This continues until the name server it wanted is found

Caching

In practice, root servers aren’t contacted much

By using a cache, a client can avoid asking external DNS servers too much

  • DNS records typically have a TTL between 1 and 24 hours
  • There’s a tradeoff between DNS load and info timelines

Anycast

Anycast is an exception to the idea of every device having a unique IP address

It allows a client to reach the nearest instance of a service

  • This implies that you can advertise the same IP (or IP block) at multiple points on the internet

Routers learn of the nearest instance topologically via the routing system

  • This means you’ll access a different instance depending on where you are

Public DNS Servers

Usually, DNS name servers should only answer external queries for their own internal domains.

There are some open public servers, though - careful about who’s seeing you make all these requests though...

  • Google public DNS is 8.8.8.8
  • Cloudflare is 1.1.1.1 - they make guarantees about privacy.
